# 연구 번들 리페어/두뇌 import 설명서

이 문서는 GPT Pro 웹 세션에서 만든 `*_nslab_episode_bundle.md` 연구 결과를
프로젝트 두뇌 원료로 넣을 때의 운영 절차를 정리한다.

## 폴더 원칙

실전 누적용 폴더는 `research/` 아래를 쓴다.

```text
research/inbox/bundles/raw/
research/inbox/bundles/repaired/
```

`docs/pilot_bundles/`와 `docs/pilot_repairs/`는 파일럿 성공 예시 보관용이다.
새 연구를 계속 모을 때는 pilot 폴더가 아니라 `research/inbox/bundles/raw/`에 넣는다.

## 한 줄 요청

새 연구 파일을 모은 뒤 Codex에게 이렇게 요청하면 된다.

```text
research/inbox/bundles/raw에 있는 연구 번들 전부 repair 돌리고 inspect 통과 확인한 다음 두뇌에 import해줘. warehouse/brain audit까지 봐줘.
```

특정 파일만 맡길 때는 이렇게 말하면 된다.

```text
이 두 파일 repair 돌리고 두뇌에 import해줘:
research/inbox/bundles/raw/YYYYMMDD_nslab_episode_bundle.md
research/inbox/bundles/raw/YYYYMMDD_nslab_episode_bundle.md
```

## 파일을 어디에 두나

웹 세션에서 받은 원본 연구 MD는 먼저 여기에 둔다.

```text
research/inbox/bundles/raw/
```

repair 결과는 여기에 생성한다.

```text
research/inbox/bundles/repaired/
```

원본은 웹 세션에서 받은 연구 결과 그대로 보존하고, 두뇌 import는 repaired 파일로 한다.

## 전체 흐름

```text
원본 연구 MD
→ repair
→ inspect-bundle
→ import-bundle --validate --accept
→ warehouse rebuild/verify
→ brain rebuild --mode catalog --allow-catalog
→ brain audit --deep
```

## 실제 명령

단일 파일 repair:

```bash
python -m news_scalping_lab.tools.repair_research_bundle research/inbox/bundles/raw/YYYYMMDD_nslab_episode_bundle.md -o research/inbox/bundles/repaired/YYYYMMDD_nslab_episode_bundle.repaired.md
```

repaired 파일 검사:

```bash
python -m news_scalping_lab.cli research inspect-bundle research/inbox/bundles/repaired/YYYYMMDD_nslab_episode_bundle.repaired.md
```

두뇌 원료로 import/accept:

```bash
python -m news_scalping_lab.cli research import-bundle research/inbox/bundles/repaired/YYYYMMDD_nslab_episode_bundle.repaired.md --validate --accept
```

warehouse 갱신:

```bash
python -m news_scalping_lab.cli warehouse rebuild
python -m news_scalping_lab.cli warehouse verify
```

catalog brain 갱신:

```bash
python -m news_scalping_lab.cli brain rebuild --mode catalog --allow-catalog
python -m news_scalping_lab.cli brain audit --deep
```

전체 코드 검증:

```bash
python -m ruff check src tests
python -m mypy src
python -m pytest -q
```

## 통과 기준

각 연구 번들은 최소한 아래가 통과해야 한다.

```text
inspection_status = validation_passed
validation_passed = true
import_loss_audit_passed = true
missing_source_reference_count = 0
missing_payload_reference_count = 0
sample_weight_validation_status = passed
quarantined_record_count = 0
```

import 후에는 아래가 맞아야 한다.

```text
imported = true
accepted = true
raw_record_count == normalized_record_count
raw_training_eligible_record_count == training_eligible_record_count
```

두뇌 갱신 후에는 아래가 맞아야 한다.

```text
warehouse verify passed = true
brain audit coverage_complete = true
brain audit deterministic_rebuild_verified = true
brain audit findings count = 0
```

`catalog_only = true`는 실패가 아니다. real LLM/embedding/provider를 붙이지 않은
오프라인 catalog brain 모드라는 뜻이다.

## repair가 하는 일

repair는 연구 내용을 새로 만들지 않는다.

하는 일:

```text
record id 전역 namespace 부여
front matter / manifest / direct ingest contract 정규화
record count / hash / validation report 재계산
source/fact/inference provenance 보강
known record_type으로 변환 가능한 record 정규화
학습에 부적합한 record는 보존하되 training_eligible=false 처리
sample_weight 합계 검증이 닫히도록 정규화
```

하지 않는 일:

```text
새 종목 지식 추가
새 수혜 관계 추가
뉴스-가격 판단을 새로 창작
가격 결과를 바꿔치기
연구 내용을 임의로 풍부하게 조작
```

## training eligible의 의미

`record_count`는 보존된 전체 두뇌 record 수다.

`training_eligible_record_count`는 그중 학습 원료로 바로 써도 되는 record 수다.

예:

```text
20241203: 341 records / 210 training eligible
20241111: 458 records / 295 training eligible
```

나머지는 버린 것이 아니다. audit, context, provenance, 검증 기록으로 남는다.
다만 근거가 덜 닫혔거나 importer가 학습 타입으로 보지 않는 record는 학습 샘플에서 제외한다.

## sample weight의 의미

`sample_weight`는 매매 비중이 아니라 학습 샘플 비중이다.

같은 날짜/같은 종목/같은 사건 묶음에 record가 여러 개 있으면 전부 `1.0`으로 두면
그 종목 하루가 과대표현된다. 그래서 한 그룹의 합계가 1.0이 되게 나눈다.

예:

```text
3개 record 그룹 = 0.333333 + 0.333333 + 0.333334 = 1.0
```

이 처리는 연구 내용을 바꾸는 것이 아니라 학습용 포장 균형을 맞추는 것이다.

## 실패했을 때 판단

바로 연구를 버리지 않는다.

먼저 아래를 구분한다.

```text
포장 실패: repair/manifest/count/type/sample_weight 문제
연구 실패: BLIND 누락, 가격 선조회, provenance 부재, source closure 실패
```

포장 실패는 repair 스크립트로 고칠 수 있다. 연구 실패는 해당 연구 번들을 두뇌 학습
원료로 쓰면 안 된다.

## 좋은 상태의 의미

좋은 상태는 아래처럼 말할 수 있다.

```text
repaired 통과
inspect 통과
import/accept 성공
quarantine 0
warehouse verify 통과
brain rebuild/audit 통과
record 손실 0
```

이 상태면 해당 연구 번들은 프로젝트의 장기 두뇌 원료로 들어간 것이다.
